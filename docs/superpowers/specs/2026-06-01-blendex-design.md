# BlendeX Design Spec

Date: 2026-06-01

## Summary

BlendeX is a two-sided integration between the CodeX APP and Blender. Users install a Blender add-on locally and add a CodeX-side plugin in the CodeX APP. The user describes a procedural creation goal in natural language; CodeX turns that goal into structured Geometry Nodes operations; the Blender add-on validates and executes those operations inside the local Blender scene.

The MVP focuses on direct, low-level Geometry Nodes graph creation and limited safe modification. It does not execute arbitrary AI-generated Python. Every Blender mutation goes through an allowlisted structured operation such as creating a node, setting a socket value, linking sockets, adding a Geometry Nodes modifier, or inspecting a graph.

## Goals

- Let CodeX create and modify Blender Geometry Nodes networks through natural language.
- Keep Blender scene changes explicit, observable, and reversible where practical.
- Make the integration work through a local WebSocket connection between the CodeX plugin and the Blender add-on.
- Let CodeX discover real capabilities from the user's installed Blender version.
- Give CodeX semantic knowledge about Geometry Nodes so it can plan useful node graphs instead of only seeing raw socket metadata.
- Provide a lightweight Blender UI for connection control and operation debugging.

## Non-Goals

- The MVP will not execute arbitrary Python generated by CodeX.
- The MVP will not attempt unrestricted editing or refactoring of any existing user-created complex Geometry Nodes graph.
- The MVP will not include a chat interface inside Blender; conversation happens in the CodeX APP.
- The MVP will not prioritize fixed high-level generators for architecture, terrain, or products. Domain examples can be built on top of the low-level tool protocol later.

## Chosen Approach

BlendeX uses the Split Bridge architecture:

- The CodeX-side plugin exposes tools, stores semantic Geometry Nodes guidance, talks to Blender over WebSocket, and translates CodeX tool calls into protocol messages.
- The Blender add-on owns the local scene, scans runtime capabilities, validates requested operations, executes safe `bpy` operations, and streams operation results and logs back to CodeX.

The core design follows a low-level operation protocol, with a small convenience layer for common setup actions. The convenience layer can create a carrier object, create or attach a Geometry Nodes modifier, initialize a clean node tree, and mark BlendeX-owned node groups. It must not become a template-heavy generator system in the MVP.

## Architecture

Runtime components:

1. CodeX APP
   - Receives the user's natural language request.
   - Plans with the available BlendeX tools.
   - Interprets operation results and errors.
   - Explains progress and can retry with corrected operations.

2. BlendeX CodeX Plugin
   - Registers structured tools for CodeX.
   - Maintains the WebSocket client connection to Blender.
   - Stores semantic node knowledge and common Geometry Nodes patterns.
   - Merges Blender runtime capability data with semantic guidance.

3. Blender Add-on
   - Provides a WebSocket server running inside Blender.
   - Scans available node types, socket templates, Blender version, current scene, selected object, modifiers, and node trees.
   - Maps structured operations to safe `bpy` calls.
   - Maintains operation history and debug logs.
   - Displays connection and operation status in Blender.

## Component Design

### CodeX Tool Registry

Defines tool schemas for CodeX. Initial tool families:

- Scene tools: inspect scene, get selected object, create carrier mesh, list objects, list modifiers.
- Capability tools: get Blender version, scan Geometry Nodes capabilities, list supported BlendeX operations.
- Graph tools: create Geometry Nodes modifier, inspect node tree, add node, link sockets, set socket value, label node, mark BlendeX ownership.
- Safety tools: validate operation batch, dry run, undo last BlendeX operation batch, check ownership before mutation.

### Semantic Node Catalog

Stores CodeX-side guidance for common Geometry Nodes concepts:

- Geometry flow, fields, attributes, instances, curves, meshes, materials, randomization, distribution, transformation, and grouping.
- Common node roles such as "scatter points", "instance geometry", "realize instances", "set material", "join geometry", and "group inputs".
- Recommended planning patterns for readable graphs.

The catalog complements runtime scanning. It does not override the user's actual Blender capabilities.

### WebSocket Transport

The MVP uses a persistent local WebSocket connection. It supports:

- Connection status and handshake.
- Session token or local authorization code.
- Request and response IDs.
- Result messages.
- Error messages.
- Operation log events.
- Capability snapshot events.

### Blender Geometry Nodes Executor

The executor validates and applies operations. It is the only module allowed to mutate Blender data. It must:

- Check object identifiers before mutation.
- Check modifier existence and type.
- Check node type availability.
- Check socket names, directions, and value types.
- Check link compatibility before creating links.
- Respect BlendeX ownership rules for modifying existing node trees.
- Return created object, modifier, node, socket, and link identifiers where possible.

### Runtime Capability Scanner

The scanner reports real local state:

- Blender version.
- Available Geometry Nodes node types.
- Node input and output socket templates.
- Socket value types.
- Current scene objects.
- Selected object.
- Existing Geometry Nodes modifiers.
- Existing node groups and whether they are BlendeX-owned.

### Blender Control and Debug Panel

The Blender panel should include:

- Start and stop controls for the BlendeX local service.
- Connection status.
- Port and session token display.
- Current CodeX connection indicator.
- Recent operation list.
- Per-operation success or failure details.
- Last error message and retry hint.

The panel will not include a natural language chat box in the MVP.

## Data Flow

1. The user describes a procedural creation goal in the CodeX APP.
2. CodeX asks BlendeX for current scene state and runtime capabilities.
3. CodeX plans a sequence of structured operations.
4. The CodeX plugin sends operations to the Blender add-on over WebSocket.
5. The Blender add-on validates each operation.
6. Valid operations are executed through the Geometry Nodes executor.
7. Blender streams back results, created IDs, logs, graph snapshots, or structured errors.
8. CodeX uses the response to continue, correct a failed step, or explain the result to the user.

## Protocol Shape

Every request uses a stable ID, operation type, target, and parameters:

```json
{
  "id": "req_1042",
  "type": "geometry_nodes.create_node",
  "target": {
    "object_id": "Cube",
    "modifier_id": "BlendeX Geometry"
  },
  "params": {
    "node_type": "GeometryNodeInstanceOnPoints",
    "label": "Instance Plants",
    "location": [420, 80]
  }
}
```

Successful response:

```json
{
  "id": "req_1042",
  "ok": true,
  "result": {
    "node_id": "node_17",
    "node_type": "GeometryNodeInstanceOnPoints"
  }
}
```

Failed response:

```json
{
  "id": "req_1042",
  "ok": false,
  "error": {
    "code": "NODE_TYPE_NOT_FOUND",
    "message": "This Blender runtime does not expose GeometryNodeInstanceOnPoints.",
    "retry_hint": "Refresh capabilities and choose an available instance-related node type."
  }
}
```

## Object and Node Tree Scope

The MVP supports both selected-object editing and new carrier object creation:

- If the user has selected an appropriate object, BlendeX should prefer that object.
- If no suitable object exists, CodeX may call a structured tool to create a new carrier mesh object.
- If the selected object has a BlendeX-owned Geometry Nodes modifier, BlendeX can modify it.
- If the selected object has a user-created complex Geometry Nodes network, BlendeX can inspect it and perform only limited, explicit, safe changes.
- BlendeX-created modifiers, node groups, and important nodes should be marked for future ownership checks.

## Safety Rules

- No arbitrary Python execution.
- Only allowlisted structured operations can mutate Blender.
- Operations must validate targets, node types, socket names, socket directions, link compatibility, and value types before mutation.
- BlendeX must not silently overwrite user-created node trees.
- Any destructive or replacement-style action requires explicit ownership or a structured confirmation path.
- Operation batches should be grouped for debug visibility and undo support.
- The Blender panel must show recent operations and failures.

## Error Handling

Errors should be both human-readable and machine-readable. Initial error codes:

- `BLENDER_NOT_CONNECTED`
- `AUTH_REQUIRED`
- `UNSUPPORTED_OPERATION`
- `OBJECT_NOT_FOUND`
- `OBJECT_NOT_SELECTED`
- `MODIFIER_NOT_FOUND`
- `NODE_TREE_NOT_FOUND`
- `NODE_TYPE_NOT_FOUND`
- `SOCKET_NOT_FOUND`
- `SOCKET_TYPE_MISMATCH`
- `LINK_NOT_ALLOWED`
- `VALUE_TYPE_MISMATCH`
- `OWNERSHIP_REQUIRED`
- `VALIDATION_FAILED`
- `EXECUTION_FAILED`

Each error should include:

- `code`
- `message`
- `retry_hint` when CodeX can plausibly recover
- optional `details` for debugging

## Testing Strategy

### Unit Tests

- Validate request and response schemas.
- Validate operation type routing.
- Validate error code mapping.
- Validate semantic catalog merge behavior.
- Validate graph operation dry-run logic without launching Blender.

### Blender Integration Tests

Run Blender in background mode and verify:

- Add-on loads successfully.
- WebSocket service starts.
- Capability scanner returns Blender version and node data.
- A mesh object can receive a Geometry Nodes modifier.
- Nodes can be created in a node tree.
- Socket values can be set.
- Compatible sockets can be linked.
- The resulting node tree matches the expected structure.

### Manual Acceptance Test

A user can:

1. Install the Blender add-on.
2. Add the CodeX-side BlendeX plugin.
3. Start the Blender local service.
4. Connect CodeX to Blender over WebSocket.
5. Ask CodeX to create a simple procedural node network.
6. See the result appear on the selected or newly created object.
7. Inspect each operation and any errors in the Blender debug panel.

## MVP Success Criteria

The MVP is successful when natural language in the CodeX APP can reliably produce visible, inspectable Geometry Nodes changes in Blender through structured operations only. The user should be able to understand what happened, see failures clearly, and trust that BlendeX will not rewrite unrelated complex node networks without explicit ownership or confirmation.

## Future Extensions

- WebSocket live preview and richer progress streaming.
- More semantic node recipes for architecture, terrain, products, scattering, and procedural materials.
- Optional high-level graph composition tools built on top of the low-level operation protocol.
- Visual graph diff before applying changes.
- Export and import of structured graph plans.
- A marketplace-ready packaging flow for both the Blender add-on and CodeX plugin.
