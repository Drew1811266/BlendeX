# BlendeX v0.5 Geometry Nodes Reasoning Design

Date: 2026-06-19

Status: Accepted target for v0.5 execution

## Goal

BlendeX v0.5 upgrades the plugin from a fixed recipe executor into a Geometry Nodes reasoning system.
Codex should be able to interpret a procedural modeling request, reason backwards
from the requested effect, choose compatible Geometry Nodes, compose a typed node
graph, validate it in Blender, repair common graph errors, and explain why the
graph was built that way.

The v0.5 bar is not "more templates". Existing recipes may remain for compatibility,
examples, and regression tests, but the primary v0.5 path must synthesize graphs
from node semantics, graph constraints, and runtime validation.

## Current Baseline

BlendeX v0.4 is ready for local beta:

- The Blender add-on can run a local authenticated service.
- The Codex-side MCP plugin can plan supported recipe workflows.
- Dry-run, confirm, inspect, batch history, and undo are implemented.
- Six fixed recipes are supported.
- Real Blender smoke validates representative architecture and scattering batches.

The v0.4 limitation is explicit: open-ended Geometry Nodes authoring is not supported.
The current planner maps prompt keywords to known recipe ids, then recipe builders emit
prewritten operation batches. This is useful infrastructure, but it is not procedural
modeling reasoning.

## Research Notes

The Blender manual organizes Geometry Nodes around node types, fields, attributes,
instances, node groups, and zones:

- Geometry Nodes index:
  https://projects.blender.org/blender/blender-manual/raw/branch/main/manual/modeling/geometry_nodes/index.rst
- Fields:
  https://projects.blender.org/blender/blender-manual/raw/branch/main/manual/modeling/geometry_nodes/fields.rst
- Attributes:
  https://projects.blender.org/blender/blender-manual/raw/branch/main/manual/modeling/geometry_nodes/attributes_reference.rst
- Instances:
  https://projects.blender.org/blender/blender-manual/raw/branch/main/manual/modeling/geometry_nodes/instances.rst
- Simulation Zone:
  https://projects.blender.org/blender/blender-manual/raw/branch/main/manual/modeling/geometry_nodes/simulation/simulation_zone.rst

Important implications:

- Fields are functions evaluated in a data-flow context, not ordinary scalar values.
- Attribute types and domains determine whether values can be used correctly.
- Instances are efficient procedural geometry, but some downstream operations require
  realized geometry.
- Zones impose graph boundary rules and version-specific behavior.
- A planner must reason about sockets, data flow, field context, domains, instance
  state, and final output connectivity.

The BlenderAlchemy paper argues for searching an action space using a generator and
state evaluator rather than relying on a single generated answer:
https://arxiv.org/abs/2404.17672

Self-Refine supports iterative generate-feedback-refine loops at test time:
https://arxiv.org/abs/2303.17651

For BlendeX, the practical architecture is:

1. Generate a graph candidate from semantic node knowledge.
2. Validate it locally as typed IR.
3. Build it in Blender in a safe/provisional way.
4. Inspect concrete Blender state and errors.
5. Repair the graph with bounded retries.

## Definition of "Understanding"

For v0.5, BlendeX demonstrates Geometry Nodes understanding when all of these are
true:

- It can describe what a supported node does using structured semantics, not only a
  display name.
- It can distinguish geometry data flow, field sockets, values, attributes, domains,
  instances, realized geometry, and final graph output.
- It can infer node candidates from an effect goal, such as scatter, deform, select,
  capture, join, realize, or assign material.
- It can connect nodes by compatible socket constraints.
- It can reject impossible or unsupported requests with a specific missing capability.
- It can explain the selected graph in terms of effects and constraints.
- It can pass held-out prompts that do not map to a fixed recipe id.

## Supported v0.5 Scope

Deep reasoning support targets the core procedural modeling domains that matter most
for a usable beta:

- Architecture and modular construction.
- Scattering and instancing.
- Procedural deformation and variation.
- Selection masks, fields, attributes, and material assignment.

The semantic catalog should cover every runtime-discovered node at a shallow level
when possible, and at least 80 to 120 high-value Geometry Nodes/Function/Shader nodes
at a deep level.

## Non-Goals

- No claim of correct arbitrary Geometry Nodes generation for every Blender node.
- No free-form execution of arbitrary Python generated by the model.
- No marketplace release bar.
- No photoreal asset generation requirement.
- No full simulation-system synthesis beyond constrained recognition and rejection.
- No template-only expansion as the v0.5 success path.

## Architecture

### Runtime Capability Layer

The Blender add-on scans available node types and returns a normalized description:

- Blender version.
- Node identifier and display name.
- Input/output sockets with identifiers, names, socket idnames, default values, enum
  candidates, field capability, multi-input behavior, and optional subtype metadata.
- Version and metadata completeness flags.

### Node Knowledge Layer

The Codex-side plugin merges runtime capabilities with curated semantics:

- Node role and effect tags.
- Preconditions and postconditions.
- Input/output semantic roles.
- Field and domain behavior.
- Instance/realization behavior.
- Typical pairings and anti-pairings.
- Repair hints.
- Documentation source pointers.

### Intent Layer

User prompts are normalized into effect intents:

- Desired primary effect.
- Secondary effects.
- Constraints and parameters.
- Scene/object assumptions.
- Required graph output.
- Unsupported or ambiguous requirements.

### Graph Planning IR

The graph IR is independent of the operation batch format. It models:

- Nodes and sockets.
- Constant values.
- Links.
- Geometry flow state.
- Field/value state.
- Attribute domain/type state.
- Instance vs realized geometry state.
- Exposed group inputs.
- Explanation traces.

The IR exports to the existing safe operation batch only after validation.

### Planner

The planner performs bounded search:

- Start from the desired final geometry/effect.
- Select candidate operators by semantic postconditions.
- Satisfy preconditions recursively.
- Unify sockets and value types.
- Score graphs for simplicity, compatibility, determinism, and user intent fit.
- Prefer generated graphs over fixed recipe ids for v0.5 requests.

### Validation and Repair

Validation has two layers:

- Static validation over graph IR.
- Runtime validation in Blender through dry-run/provisional build/inspect.

Repair handles common issues:

- Missing final output.
- Missing source geometry.
- Socket name drift across Blender versions.
- Field/value mismatch.
- Missing realize instances before mesh-only operations.
- Invalid links.
- Unsupported node type.

## Anti-Template Requirement

The v0.5 readiness audit must run tests with the fixed recipe registry disabled or
ignored. These tests must pass:

- Novel held-out prompts produce graph IR.
- Generated graphs contain node choices justified by semantic effects.
- No prompt result is accepted merely because it matched a recipe id.
- At least 20 held-out prompts produce valid operation batches.
- At least 8 held-out prompts pass real Blender smoke when Blender is available.

Templates may exist only as:

- Compatibility shims for v0.4 recipes.
- Examples for documentation.
- Regression fixtures.
- Small semantic motifs/operators that still require graph planning to compose.

## Final v0.5 Acceptance Criteria

- Unit and integration-style test suite passes.
- Real Blender smoke passes for representative generated graphs.
- The benchmark demonstrates graph synthesis without recipe ids.
- The planner explains node choices and graph structure.
- Unsupported requests identify missing capabilities.
- Fixed recipes are no longer the primary open-ended planning mechanism.
- Version metadata and packaging are synchronized for 0.50.0.
